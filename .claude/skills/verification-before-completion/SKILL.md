# 完了前の検証

## 概要

検証なしに作業完了を宣言するのは、効率性ではなく不誠実さです。

**核心原則:** 常に証拠を先に、主張は後から。

**このルールの文言に違反することは、精神に違反することと同じです。**

## 鉄則

```
新たな検証証拠なしに完了を宣言してはならない
```

このメッセージで検証コマンドを実行していなければ、通過したと主張することはできません。

## ゲート関数

```
任意のステータスを主張したり満足感を表現する前に:

1. 特定: この主張を証明するコマンドは何か？
2. 実行: 完全なコマンドをフルで実行（新鮮に、完全に）
3. 読む: 出力全体を確認し、終了コードを確認し、失敗数を数える
4. 検証: 出力は主張を確認しているか？
   - NO の場合: 証拠とともに実際のステータスを述べる
   - YES の場合: 証拠とともに主張する
5. その後のみ: 主張を行う

いずれかのステップをスキップ = 検証ではなく嘘
```

## よくある失敗

| 主張 | 必要なもの | 不十分なもの |
|-------|----------|----------------|
| テストがパスする | テストコマンド出力: 0 failures | 以前の実行、「パスするはず」 |
| Linter がクリーン | Linter 出力: 0 errors | 部分チェック、推測 |
| ビルドが成功する | ビルドコマンド: exit 0 | Linter 通過、ログが良さそう |
| バグが修正された | 元の症状をテスト: passes | コードを変更した、修正したと仮定 |
| 回帰テストが動く | RED-GREEN サイクルを確認 | テストが一度パスした |
| エージェントが完了した | VCS の差分に変更が見える | エージェントの「成功」報告 |
| 要件が満たされた | 項目ごとのチェックリスト | テストがパスしている |

## 危険なサイン — STOP

- 「〜のはず」「おそらく」「〜のようだ」を使っている
- 検証前に満足感を表現している（「いいですね！」「完璧です！」「完了です！」など）
- 検証なしにコミット・push・PR を作ろうとしている
- エージェントの成功報告を信頼している
- 部分的な検証に頼っている
- 「今回だけ」と考えている
- **検証を実行せずに成功を示唆するあらゆる言い回し**

## 合理化の防止

| 言い訳 | 現実 |
|--------|---------|
| 「今は動くはず」 | 検証を実行してください |
| 「自信がある」 | 自信 ≠ 証拠 |
| 「今回だけ」 | 例外なし |
| 「Linter がパスした」 | Linter ≠ コンパイラ |
| 「エージェントが成功と言った」 | 独立して検証してください |
| 「部分チェックで十分」 | 部分は何も証明しない |

## 主要パターン

**テスト:**
```
✅ [テストコマンドを実行] [34/34 パスを確認] 「全テストがパスします」
❌ 「今は通るはず」 / 「正しく見える」
```

**ビルド:**
```
✅ [ビルドを実行] [exit 0 を確認] 「ビルドがパスします」
❌ 「Linter がパスした」
```

**要件:**
```
✅ 計画を再読 → チェックリスト作成 → 各項目を検証 → 報告
❌ 「テストがパスする、フェーズ完了」
```

**エージェント委任:**
```
✅ エージェントが成功を報告 → VCS の差分を確認 → 変更を検証 → 実際のステータスを報告
❌ エージェントの報告を信頼する
```

## 結論

**検証のショートカットはありません。**

コマンドを実行してください。出力を読んでください。その後で結果を主張してください。

これは交渉の余地がありません。

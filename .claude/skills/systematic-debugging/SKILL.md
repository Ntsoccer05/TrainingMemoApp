# 体系的デバッグ

## 概要

場当たり的な修正は時間を無駄にし、新たなバグを生み出します。応急処置は根本的な問題を隠します。

**核心原則：** 修正を試みる前に必ず根本原因を見つけること。症状への対処は失敗です。

**このプロセスの文言に違反することは、デバッグの精神に違反することと同じです。**

## 鉄則

```
根本原因の調査なしに修正を提案してはならない
```

フェーズ1を完了していなければ、修正を提案できません。

## 使うタイミング

あらゆる技術的な問題に使用します：
- テストの失敗
- 本番環境のバグ
- 予期しない動作
- パフォーマンス問題
- ビルドの失敗
- 統合の問題

**特に以下の場合は必ず使用してください：**
- 時間的プレッシャーがある（緊急時こそ推測が誘惑的になる）
- 「たった一つの簡単な修正」が明白に見える
- すでに複数の修正を試みた
- 前の修正が機能しなかった
- 問題を完全に理解していない

**スキップしてはいけない場合：**
- 問題が単純に見える（単純なバグにも根本原因がある）
- 急いでいる（急ぐと手戻りが保証される）
- 今すぐ直してほしいと言われている（体系的な方が場当たりより速い）

## 4つのフェーズ

次のフェーズに進む前に各フェーズを完了しなければなりません。

### フェーズ1：根本原因の調査

**修正を試みる前に：**

1. **エラーメッセージを注意深く読む**
   - エラーや警告を読み飛ばさない
   - 多くの場合、正確な解決策が含まれている
   - スタックトレースを完全に読む
   - 行番号、ファイルパス、エラーコードをメモする

2. **一貫して再現する**
   - 確実に発生させられるか？
   - 正確な手順は？
   - 毎回発生するか？
   - 再現できない場合 → データを収集する、推測しない

3. **最近の変更を確認する**
   - これを引き起こし得る変更は何か？
   - git diff、最近のコミット
   - 新しい依存関係、設定変更
   - 環境の違い

4. **複数コンポーネントシステムで証拠を収集する**

   **複数コンポーネントを持つシステムの場合（CI → ビルド → 署名、API → サービス → データベース）：**

   **修正を提案する前に、診断のための計装を追加する：**
   ```
   各コンポーネント境界について：
     - コンポーネントに入るデータをログに記録する
     - コンポーネントから出るデータをログに記録する
     - 環境/設定の伝播を確認する
     - 各レイヤーでの状態を確認する

   一度実行してどこで壊れるかを示す証拠を収集する
   それから証拠を分析して失敗しているコンポーネントを特定する
   それから特定のコンポーネントを調査する
   ```

   **例（多層システム）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **これが明らかにするもの：** どのレイヤーで失敗するか（secrets → workflow ✓、workflow → build ✗）

5. **データフローをトレースする**

   **エラーがコールスタックの奥深くにある場合：**

   このディレクトリの `root-cause-tracing.md` を参照してください。

   **クイックバージョン：**
   - 不正な値の発生源はどこか？
   - 何がこれに不正な値を渡したか？
   - ソースを見つけるまで遡る
   - 症状ではなくソースで修正する

### フェーズ2：パターン分析

**修正する前にパターンを見つける：**

1. **動作している例を見つける**
   - 同じコードベースで似た動作しているコードを探す
   - 壊れているものに似て動作しているものは何か？

2. **リファレンスと比較する**
   - パターンを実装している場合は、リファレンス実装を完全に読む
   - ざっと読まない — 一行ずつ読む
   - 適用する前にパターンを完全に理解する

3. **差異を特定する**
   - 動作しているものと壊れているものの差異は何か？
   - すべての差異をリストアップする（どんなに小さくても）
   - 「それは関係ないはず」と決めつけない

4. **依存関係を理解する**
   - これが必要とする他のコンポーネントは何か？
   - どんな設定、構成、環境が必要か？
   - どんな前提を置いているか？

### フェーズ3：仮説と検証

**科学的手法：**

1. **単一の仮説を立てる**
   - 明確に述べる：「XがYという理由で根本原因だと考える」
   - 書き留める
   - 曖昧ではなく具体的に

2. **最小限にテストする**
   - 仮説を検証するために可能な最小限の変更を行う
   - 一度に一つの変数
   - 複数のことを同時に修正しない

3. **続ける前に確認する**
   - 機能した？Yes → フェーズ4
   - 機能しなかった？新しい仮説を立てる
   - 上に修正を追加しない

4. **わからない場合**
   - 「Xが理解できていない」と言う
   - 知っているふりをしない
   - 助けを求める
   - もっと調査する

### フェーズ4：実装

**症状ではなく根本原因を修正する：**

1. **失敗するテストケースを作成する**
   - 可能な限りシンプルな再現
   - 可能であれば自動テスト
   - フレームワークがない場合は一時的なテストスクリプト
   - 修正する前に必ず作成する
   - 適切な失敗するテストの作成には `superpowers:test-driven-development` スキルを使用

2. **単一の修正を実装する**
   - 特定した根本原因に対処する
   - 一度に一つの変更
   - 「ついでに」改善しない
   - まとめたリファクタリングをしない

3. **修正を確認する**
   - テストは今パスするか？
   - 他のテストは壊れていないか？
   - 問題は実際に解決されたか？

4. **修正が機能しない場合**
   - 停止する
   - カウント：何回修正を試みたか？
   - 3回未満：フェーズ1に戻り、新しい情報で再分析
   - **3回以上：停止してアーキテクチャに疑問を持つ（下記ステップ5）**
   - アーキテクチャの議論なしに修正#4を試みない

5. **3回以上失敗した場合：アーキテクチャに疑問を持つ**

   **アーキテクチャ問題を示すパターン：**
   - 各修正が異なる場所で新たな共有状態/結合/問題を露わにする
   - 修正が「大規模なリファクタリング」を必要とする
   - 各修正が他の場所に新たな症状を生み出す

   **停止して根本的な問題に疑問を持つ：**
   - このパターンは本質的に健全か？
   - 「単なる惰性でこれを続けている」か？
   - 症状の修正を続けるのではなくアーキテクチャをリファクタリングすべきか？

   **これは仮説の失敗ではない — これは間違ったアーキテクチャです。**

## 危険なサイン — 停止してプロセスに従う

以下のことを考えていたら：
- 「今は応急処置、後で調査」
- 「ただXを変えて動くか見てみよう」
- 「複数の変更を追加してテストを実行」
- 「テストをスキップして手動で確認しよう」
- 「おそらくXだろう、それを修正しよう」
- 「完全には理解していないがこれで動くかもしれない」
- 「パターンはXと言っているが別の方法で適用しよう」
- 「主な問題は：[調査なしに修正をリスト]」
- データフローをトレースする前に解決策を提案する
- **「もう一度修正を試みよう」（すでに2回以上試みた場合）**
- **各修正が異なる場所に新たな問題を露わにする**

**これらすべては：停止。フェーズ1に戻ること。を意味します。**

**3回以上失敗した場合：** アーキテクチャに疑問を持つ（フェーズ4.5参照）

## 間違ったアプローチのシグナル

**以下のリダイレクションに注意してください：**
- 「それは起きていないのでは？」— 確認せずに仮定した
- 「〜を確認してくれませんか？」— 証拠収集を追加すべきだった
- 「推測するのをやめて」— 理解せずに修正を提案している
- 「これについて深く考えて」— 症状だけでなく根本的な問題を疑う
- 「詰まっていますか？」（不満気に）— アプローチが機能していない

**これらを見たら：** 停止。フェーズ1に戻る。

## よくある合理化

| 言い訳 | 現実 |
|--------|---------|
| 「問題が単純なのでプロセスは不要」 | 単純な問題にも根本原因がある。プロセスは単純なバグには速い。 |
| 「緊急なのでプロセスに時間はない」 | 体系的デバッグは場当たりより速い。 |
| 「まずこれを試してから調査しよう」 | 最初の修正がパターンを設定する。最初から正しくやる。 |
| 「修正が機能することを確認した後でテストを書く」 | テストされていない修正は定着しない。テストを先に書くことが証明になる。 |
| 「一度に複数の修正で時間節約」 | 何が機能したかを分離できない。新たなバグを引き起こす。 |
| 「リファレンスが長すぎるのでパターンを適用する」 | 不完全な理解がバグを保証する。完全に読む。 |
| 「問題が見える、修正しよう」 | 症状を見ること ≠ 根本原因を理解すること。 |
| 「もう一度修正を試みよう」（2回以上失敗後） | 3回以上の失敗 = アーキテクチャ問題。パターンを疑う、再び修正しない。 |

## クイックリファレンス

| フェーズ | 主な活動 | 成功基準 |
|-------|---------------|------------------|
| **1. 根本原因** | エラーを読む、再現する、変更を確認する、証拠を収集する | 何が・なぜを理解する |
| **2. パターン** | 動作している例を見つける、比較する | 差異を特定する |
| **3. 仮説** | 理論を立てる、最小限にテストする | 確認されるか新しい仮説 |
| **4. 実装** | テストを作成、修正、確認 | バグ解決、テストパス |

## プロセスで「根本原因なし」が明らかになった場合

体系的な調査で問題が本当に環境的、タイミング依存、または外部依存であることが明らかになった場合：

1. プロセスを完了した
2. 何を調査したかを文書化する
3. 適切な処理を実装する（リトライ、タイムアウト、エラーメッセージ）
4. 将来の調査のための監視/ログを追加する

**ただし：** 「根本原因なし」ケースの95%は不完全な調査です。

## サポートテクニック

これらのテクニックは体系的デバッグの一部であり、このディレクトリで利用できます：

- **`root-cause-tracing.md`** — コールスタックを遡ってバグをトレースし、元のトリガーを見つける
- **`defense-in-depth.md`** — 根本原因発見後に複数レイヤーでバリデーションを追加する
- **`condition-based-waiting.md`** — 任意のタイムアウトを条件ベースのポーリングに置き換える

**関連スキル：**
- **superpowers:test-driven-development** — 失敗するテストケースの作成（フェーズ4、ステップ1）
- **superpowers:verification-before-completion** — 修正が機能したことを主張する前に確認する

## 実世界への影響

デバッグセッションから：
- 体系的アプローチ：修正まで15〜30分
- 場当たりアプローチ：2〜3時間の試行錯誤
- 一回での修正成功率：95% vs 40%
- 新たに導入されたバグ：ゼロに近い vs 一般的

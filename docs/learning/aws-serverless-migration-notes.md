## 1. AWSアカウントとIAM(権限管理)の基礎

### root ユーザーとIAMユーザー・IAMロールの違い

AWSアカウントを作った直後にログインできるのが **rootユーザー** です。rootはそのアカウントの全操作ができる「万能鍵」なので、AWSは公式に「rootは初期セットアップ以外では使わない」ことを推奨しています。

今回の調査で分かったこと:
- trainingMemoアカウントのrootには **アクセスキーが存在しない**(良い状態。コンソールログインのみで、プログラムからの直接操作はできない)
- ただし **IAMユーザーもIAMロールも0件** で、これまでずっとroot(コンソールログイン)だけで運用されてきたことが判明

日常的な運用や自動化(Terraform、CI/CD)では、root本人ではなく以下を使うのがベストプラクティスです。

| 種類 | 用途 | 特徴 |
|---|---|---|
| **IAMユーザー** | 人間が長期的に使うアカウント | ユーザー名+パスワード or アクセスキーを持つ。今回は「ローカルからTerraformを実行する人」用に1つ作成 |
| **IAMロール** | 一時的に権限を借りる仕組み | 長期の秘密情報を持たず、「引き受ける(Assume)」ことで一時的な権限を得る。GitHub ActionsやEC2/Lambdaが使う権限はロールにするのが基本 |
| **OIDC federation** | 外部サービス(GitHub等)にIAMロールを安全に貸し出す仕組み | GitHub Actions用に長期アクセスキーを発行せず、実行のたびに短期トークンでロールを借りる。漏洩リスクが大幅に下がる |

今回のTerraform実装(`infra/bootstrap/`)では、GitHub Actions用に **OIDC + IAMロール**、ローカル作業用に **MFA必須のIAMユーザー** を新規作成し、rootは今後一切使わない構成にしました。

### MFA(多要素認証)

パスワードだけでなく、スマートフォンやセキュリティキーでの追加認証を必須にする仕組み。trainingMemoのrootは既にMFA(パスキー/セキュリティキー)が設定済みでした。今回作成するローカル操作用IAMユーザーにも「MFAなしでは何もできない」ポリシーを付けています。

## 2. ネットワークの基礎(VPC・サブネット)

### VPCとは

VPC(Virtual Private Cloud)は、AWS上に作る「自分専用の仮想ネットワーク」です。trainingMemoアカウントには `trainingMemo-vpc`(CIDR `10.0.0.0/16`)が1つあります。CIDR表記(`10.0.0.0/16`)は「使えるIPアドレスの範囲」を表し、`/16`は約6万5千個のIPアドレスを持つ広い範囲という意味です。

### パブリックサブネットとプライベートサブネット

VPCの中はさらに「サブネット」という小さい区画に分かれています。trainingMemoには4つありました。

- `trainingMemo-public-1a` / `-1c`: インターネットゲートウェイへの経路がある = インターネットと直接通信できる
- `trainingMemo-private-1a` / `-1c`: インターネットへの経路がない = 外部から直接アクセスできない、外部への通信もできない

「パブリック/プライベート」の違いは、そのサブネットに紐づく **ルートテーブル**(通信の行き先を決める表)に「0.0.0.0/0(＝インターネット全体)への経路」があるかどうかで決まります。今回調査したところ、trainingMemoのprivateサブネットのルートテーブルには`local`(VPC内部への経路)しかなく、インターネットへの経路が一切ありませんでした。

- EC2(アプリサーバー)は **publicサブネット** に配置(直接インターネットと通信できる必要があるため)
- RDS(データベース)は **privateサブネット** に配置(外部から直接狙われないようにするため。これは正しいベストプラクティス)

### NAT Gateway / NAT Instance

「プライベートサブネットにいるけれど、外部のAPIを呼び出したい」というケースがあります(今回でいうと、LaravelのSocialiteによるGoogleなどの外部OAuth認証)。この「プライベートサブネットから外部への一方通行の出口」を作る仕組みがNATです。

- **NAT Gateway**: AWSが管理してくれる高可用性なNATサービス。楽だが月$32程度からと高め
- **NAT Instance**: 自分でEC2インスタンス1台をNAT代わりに動かす方式。安い(月数ドル)が、可用性は自分で見る必要がある

trainingMemoは低トラフィックの個人利用アプリのため、コスト優先で **NAT Instance** を採用することにしました。

## 3. セキュリティグループ(SG)の基礎

セキュリティグループは、EC2やRDSなどの「入口・出口の門番」です。「どのポート番号を」「どこから(誰に)」許可するかをルールとして設定します。

### 今回見つかった問題と修正

trainingMemoのEC2用SG(`web-sg`)を調査したところ、以下が **`0.0.0.0/0`(＝全世界)** に開放されていました。

- SSH(22番ポート): サーバーへのリモートログイン用ポート。世界中に開いていると、ブルートフォース(パスワード総当たり)攻撃の対象になり得る
- HTTP(80)/HTTPS(443): 本来ALB(ロードバランサー)を経由すべきところ、EC2に直接アクセスできてしまう「裏道」になっていた

**セキュリティグループの送信元は、IPアドレス(CIDR)だけでなく「別のセキュリティグループ」も指定できます。** 今回は以下のように修正しました。

| ポート | 修正前 | 修正後 |
|---|---|---|
| SSH(22) | `0.0.0.0/0` | 自分の現在のIPアドレスのみ |
| HTTP(80)/HTTPS(443) | `0.0.0.0/0` | ALBのセキュリティグループ(`http-ssh-full`)からのみ |

「ALBのセキュリティグループからのみ許可」にすると、ALBを経由した正規の通信は引き続き通りますが、EC2への直接アクセスの裏道は塞がれます。これは「多層防御(EC2自体が直接インターネットに晒されないようにする)」という考え方の実践例です。

## 4. コスト構造の考え方

### 「常時稼働」と「従量課金」の違い

- EC2・RDS(Single-AZ)・ALBは、動かしていなくても(正確には「起動している限り」)時間単位で課金される「常時稼働型」の料金体系
- Lambda・API Gatewayは「呼び出された回数・時間」でしか課金されない「従量課金型」

trainingMemoのように低トラフィックなら、**同じ処理でも従量課金型の方が圧倒的に安くなる** ケースが多いです。これが今回の「サーバーレス化でコスト削減」の根本原理です。

### 見えにくい無駄なコスト

調査の過程で、EC2とRDSは(コスト削減のため)手動停止されているにもかかわらず、**ALBだけは止め忘れて課金され続けていた** ことが分かりました。ALBのような「常時稼働の付帯サービス」は、本体を止めても個別に止めない限り課金が続く点に注意が必要です。

## 5. Terraformの基礎(IaC)

### なぜコードでインフラを管理するのか

AWSコンソールでポチポチ設定する方法は、変更履歴が残らず、他人と共有しづらく、同じ環境を再現しにくいという欠点があります。Terraformは「インフラの構成をコード(`.tf`ファイル)で書き、そのコード通りにAWSへ反映する」ツールです。

### 今回学んだ主要概念

| 用語 | 意味 |
|---|---|
| **Provider** | どのクラウド(AWS/GCP/Azure等)を操作するかの設定。今回は`aws`プロバイダを使用 |
| **Resource** | 実際に作るAWSリソース1つ1つ(例: `aws_s3_bucket`, `aws_lambda_function`) |
| **Data source** | 「新規作成せず、既存のリソースを参照するだけ」のブロック(`data "aws_vpc" ...`)。今回、既存のRDS/Route53/ACMを壊さず参照するために多用 |
| **Module** | `.tf`ファイル群をひとまとめにした再利用可能な部品。今回は`frontend`・`nat_instance`・`backend_lambda`・`routing`の4モジュールに分割 |
| **State(状態ファイル)** | Terraformが「今どんなリソースが存在しているか」を記録するファイル(`.tfstate`)。チームで共有する場合はS3等に保存する |
| **Backend** | state をどこに保存するかの設定。今回はS3 + DynamoDB(複数人が同時に`apply`しないためのロック機構)を新規構築 |

### ブートストラップという考え方

「Terraformの状態を保存する場所(S3バケット)自体をTerraformで作りたい」という場合、"state を保存する場所がまだ存在しない状態で、その場所を作るコードを実行する" という鶏と卵の問題が起きます。これを避けるため、`infra/bootstrap/`だけは最初の1回だけ手動(ローカルstate)で適用し、それ以降の本体(`infra/terraform/`)は`bootstrap`で作ったS3/DynamoDBをbackendとして使う、という2段構えにしています。

## 6. これから出てくる予定の用語(実装が進み次第、随時追記)

- **Lambda / Bref**: サーバーを持たずにコードを実行する仕組み(Lambda)と、PHP/LaravelをLambdaで動かすためのOSSツール(Bref)
- **API Gateway**: インターネットからのHTTPリクエストをLambdaに橋渡しするサービス
- **CloudFront / OAC**: 世界中に配置されたキャッシュサーバー経由でコンテンツを配信するCDN。S3への直接アクセスを禁止し、CloudFront経由のみ許可する仕組みがOAC(Origin Access Control)
- **ACM(証明書)**: HTTPS通信に必要なSSL/TLS証明書を無料で発行・管理するサービス。CloudFront用の証明書は必ず`us-east-1`リージョンで発行する必要があるという特殊ルールがある

## 7. MFA付き一時セッション(STS)のつまずきポイント

Task 1完了時、`aws sts get-session-token`でMFAコードが常に`AccessDenied`になる問題が発生しました。原因は **`--serial-number`に指定したMFAデバイスのARNが実際に登録されたものと違っていた** ことでした。

IAMでMFAデバイスを新規作成すると、指定した名前がそのままデバイスの識別子(ARNの末尾)になります。今回は`trainingmemo-terraform-operator`という名前でユーザーを作ったのに、MFAデバイス自体は`trainingmemo-Authentication`という別名で登録されていたため、`--serial-number arn:aws:iam::<account>:mfa/trainingmemo-terraform-operator`という想定のARNが実在せず、コードの正誤以前に「デバイスが見つからない」状態でした。

**教訓:** MFA関連のコマンドでARNを指定する際は、IAMコンソールの対象ユーザー→「セキュリティ認証情報」タブ→「多要素認証(MFA)」欄に表示されている **実際の識別子(ARN)をコピーして使う** のが確実。名前を「こう付けたはず」で推測しないこと。

また、TOTP(6桁コード)は30秒ごとに変わるため、コマンド実行までに時間がかかると失効します。エラーが出た場合は「ARNが合っているか」と「コードが最新か」の両方を疑うとよいことも分かりました。

## 8. Terraformモジュール間の「循環参照」問題

複数のモジュールを組み合わせると、「AがBの出力を必要とし、BもAの出力を必要とする」という循環(サイクル)が起きることがあります。今回は以下の組み合わせで発生しました。

- `frontend`モジュール(S3バケット作成): CloudFrontからのアクセスだけを許可するバケットポリシーを設定するには、CloudFrontのARN(`routing`モジュールが作る)が要る
- `routing`モジュール(CloudFront作成): S3バケットの情報(`frontend`モジュールが作る)が要る

これはTerraformの依存関係グラフ上、解決不能な「AがBを待ち、BがAを待つ」状態になります。**解決策は「両方が必要とする最後の1ピース(バケットポリシー)だけを、どちらのモジュールにも属させず、それらを呼び出す側(ルートモジュール)に置く」** というものでした。

```
frontend(S3+OAC作成、ポリシーなし)
  → routing(CloudFront作成。frontendの出力を使う)
    → バケットポリシー(ルート側で定義。frontendのバケットとroutingのCloudFront ARN、両方を参照)
```

こうすると依存の向きが「frontend → routing → バケットポリシー」の一方向になり、循環が解消します。**モジュール設計で「AとBが互いを必要としている」と気づいたら、両者に依存する第三の場所(呼び出し元)に処理を移せないか考える** のが定石だと学びました。

### 危険だった副産物: 孤立した重複ファイル

上記の整理をする過程で、当初「共通設定」として作っていた`infra/terraform/providers.tf`等が、新しく作った`infra/terraform/envs/production/`と**全く同じtfstateの保存場所(S3バケット名・キー)を指したまま、誰にも読み込まれない孤立ファイル**として残ってしまいました。これは見た目には無害ですが、もし誰かが誤ってそのディレクトリで`terraform apply`を実行すると、「今保存されている本番の状態(実際に動いているリソース一覧)と、そのディレクトリの中身(空っぽ)」を比較して、**「本番リソースを全部削除する」計画が立てられ、そのまま実行されてしまう**という重大なリスクがありました。

**教訓:** 同じ`backend`設定(tfstateの保存場所)を複数のディレクトリに書いてはいけない。使われなくなった`.tf`ファイルは、動作に影響しないからと放置せず、速やかに削除する。

## 9. Lambdaは「毎回まっさらな環境」であることを忘れない

Lambda(サーバーレス関数)は、リクエストのたびに使い捨ての実行環境で動きます(コールドスタート時は特に)。このため、EC2で動かしていた頃の「当たり前」がいくつも通用しなくなります。今回の実装で実際に踏んだ落とし穴は以下の通りです。

- **セッション**: `SESSION_DRIVER=file`だと、セッション情報をローカルディスクに書くが、次のリクエストは別の実行環境かもしれないので読めない → `database`ドライバに変更
- **キャッシュ**: 同じ理由で`CACHE_DRIVER=file`も機能しない。今回、これを見落としたまま実装を進めてしまい、**最終レビューの段階で「LaravelのAPIレート制限(throttleミドルウェア)がキャッシュへの書き込みに失敗し、全APIリクエストがエラーになる」という重大な欠陥**として発覚しました。セッションとキャッシュは「ファイルに書けない」という同じ原因を持つ双子のような設定なのに、片方(セッション)だけ対応して、もう片方(キャッシュ)を見落とす、というのは初学者が陥りやすい典型パターンだと感じました。
- **`.env`ファイルはLambdaに届かない**: デプロイ用のzipにはgit管理されたファイルだけが含まれ、`.env`(gitignore対象)は含まれません。つまり、ローカル開発で`.env`に書いた設定は、Lambda上では**Terraformで明示的に「環境変数」として設定しない限り反映されません**。「`.env.example`を直したから安心」ではなく、「Lambda関数の環境変数にも同じ値を設定したか」を必ずセットで確認する必要があります。

## 10. Bref(LaravelをLambdaで動かす仕組み)の2つの顔

Brefには、大きく分けて2種類の「レイヤー」(Lambda関数に追加する機能拡張パッケージ)があります。

| レイヤー名の例 | 用途 | 今回のLambda関数 |
|---|---|---|
| `php-82-fpm` | HTTPリクエストを処理する(PHP-FPMというWebサーバーの仕組みを使う) | ブラウザ・APIからのアクセスを受けるメインの関数 |
| `php-82`(FPMなし) | コンソール(CLI)コマンドを実行する専用 | `php artisan migrate`のようなコマンドを実行する専用の別関数 |

最初、`php artisan migrate`をLambdaで実行する処理を「HTTPリクエスト処理用の関数(`php-82-fpm`)」に間違って向けてしまい、コードレビューで「そのレイヤーはコンソールコマンドを実行できない」と指摘されました。**「1つのアプリだから1つのLambda関数で全部やろう」とは考えず、Brefでは「HTTPを受ける関数」と「コマンドを実行する関数」を分けて用意する必要がある**、というのが今回の学びです(同じLaravelのコードを使い回しつつ、Lambda関数自体は2つ作る形です)。

また、Brefが呼び出す際のデータの渡し方(ペイロード)にも決まりがあり、`{"cli": "migrate --force"}`のように「artisanに渡す引数だけ」を書く必要があります。「`php artisan migrate --force`」のようにコマンド全体を書いてしまうと、二重に`php artisan`を実行しようとしてエラーになります。細かい規約ですが、公式ドキュメントの実例をよく確認することが大切だと感じました。

## 11. CloudFrontの「ルーティング漏れ」に気づく難しさ

CloudFrontは「このパス(URL)はこっちのサーバーへ」という振り分けルール(behavior)を書きますが、**Laravelのようなフレームワークは、アプリのコード上には見えない「裏側の決まったパス」を勝手に使っている**ことがあります。

今回見つかった漏れ:
- `/sanctum/csrf-cookie`: ログイン前に必ず呼ばれる認証の下準備用パス(`/api/`配下ではない)
- `/livewire/*`: Filament(管理画面)が裏側で使うAJAX通信用パス(`/admin`配下ではない)

これらを「APIサーバーへ転送する」ルールに入れ忘れると、静的サイト用のサーバー(S3)に届いてしまい、見た目上はページが表示される(SPAのフォールバック機能のおかげ)ため、**一見動いているように見えて、実は認証や管理画面の操作だけが壊れている**という気づきにくい不具合になります。「主要な画面が表示されるか」だけでなく、「ログインや管理操作まで実際に動くか」を確認する重要性を学びました。

## 12. コードレビューを2段階に分ける効果

今回の実装では、1つの作業ごとに「①仕様通りに作られているか(仕様適合レビュー)」→「②設計・セキュリティ・運用面で問題がないか(コード品質レビュー)」という2段階のチェックを行いました。実際にこの2段階目で見つかった問題には、以下のような重大なものがありました。

- セキュリティグループの過剰な公開範囲
- 本番を巻き込みかねない孤立ファイルの危険性
- ログイン・管理画面が壊れるCloudFront設定漏れ
- マイグレーション失敗がCIに検知されない(静かに失敗する)問題
- AWS認証情報が漏洩しかねないGitHub Actionsのスクリプトインジェクション

「動くコードを書けたら終わり」ではなく、**「意図通りに動くか」と「安全に運用できるか」は別の観点**であり、後者は特に見落としやすい、ということを実例を通じて学びました。

## 13. GitHub Actions(CI/CD)のセキュリティで気をつけたこと

- **`${{ steps.xxx.outputs.yyy }}`をスクリプトのコード内に直接埋め込まない**: 変数の中身(今回はterraform planの出力)に特殊な文字が混ざっていると、意図しないコードとして実行されてしまう危険があります(スクリプトインジェクション)。代わりに`env:`ブロック経由で渡し、`process.env.変数名`として読み込むのが安全な作法です。
- **`aws lambda invoke`は関数内部のエラーを検知しない**: コマンド自体は「呼び出しに成功したか」しか見ておらず、関数の中でエラーが起きても正常終了(exit code 0)を返します。実際の処理結果(レスポンス内容)を自分でチェックし、失敗していたらワークフロー自体を失敗させる必要があります。
- **`terraform apply -auto-approve`は「今この瞬間の状態」に対して再計画してから即実行する**: PRの時点でレビューした計画(plan)と、実際にマージ後適用される内容が微妙にズレる可能性があります。`plan`の結果をファイルに保存し、そのファイルを`apply`するようにすることで、「レビューした内容がそのまま適用される」ことを保証できます。
